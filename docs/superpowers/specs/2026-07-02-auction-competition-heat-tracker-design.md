# 경매 경쟁열기 추적·비교분석 도구 — 설계 문서

- 작성일: 2026-07-02
- 상태: 승인됨 (브레인스토밍 완료, 구현 계획 수립 전)
- 대상 사용자: 단일 사용자(본인). 입찰 판단 보조용 개인 도구.

## 1. 목적 / 배경

내가 입찰할 경매 물건의 **경쟁 열기(관심도)를 사전에 가늠**하고, **유사한 과거 사건들의 매각 결과와 비교**해 입찰 판단을 돕는 크롤링·집계 웹 도구를 만든다.

- 대상 경매: **법원 부동산 경매(courtauction.go.kr)** + **공매(온비드, onbid.co.kr)**
- 접근 전략: 1단계로 **선행 지표 집계 대시보드(A)**를 먼저 만들고, 데이터가 쌓이면 예측 모델(B)로 확장. 본 스펙은 A 단계 + 비교분석까지를 범위로 한다.
- 명시적 성격: 결과물은 **"보조 지표"**이며 경쟁률을 단정 예측하지 않는다(UI에 명시).

## 2. 핵심 요구사항

1. 워치리스트에 등록한 물건들의 **조회수 추세**(경쟁 열기 선행 신호)를 매일 추적한다.
2. 각 물건의 **매각 여부/결과**(매각·유찰·취하 등, 매각가, 낙찰가율)를 추적한다.
3. 내 물건과 **유사한 완료 사건들**(근처 지역·비슷한 감정가·같은 용도·유사 유찰수)을 수집해 **매각 결과를 비교 분석**한다.
4. **배포형 웹**으로 접근하되 **나만 접근**(단일 비밀번호 로그인)한다.

### 명시적 비범위(제외 사항)

- **응찰자수(입찰자 수)** 는 수집하지 않는다. 유료 사이트의 로그인/상세 페이지에서만 얻을 수 있어 "로그인 없이 목록 수준" 원칙과 충돌하기 때문. 경쟁 강도는 **조회수 + 낙찰가율**로 대체 가늠한다.
- 전국 전체 물건 자동 수집(부하·차단·매칭 리스크). 워치리스트 기반으로 한정.
- 유료 사이트 상세 페이지 크롤링. 로그인 없이 보이는 **목록/랭킹 수준**만 사용.

## 3. 데이터 소스

| 소스 | 용도 | 접근 |
|---|---|---|
| courtauction.go.kr | 법원경매 물건 기본정보·매각결과(감정가·최저가·유찰횟수·매각기일·결과·낙찰가) | 무료 공식 |
| onbid.co.kr | 공매 물건 기본정보·개찰결과 | 무료 공식 |
| 지지옥션(ggi.co.kr) | 조회수, 완료사건 낙찰가율(로그인 없이 목록 수준) | 유료사이트, 저빈도 |
| 탱크옥션(tankauction.com) | 조회수 | 유료사이트, 저빈도 |
| 두인경매(dooinauction.com) | 조회수(조회수 랭킹 제공) | 유료사이트, 저빈도 |

- 조회수 소스는 **어댑터(플러그인) 구조**로 두어, 향후 옥션원·스피드옥션 등 추가 시 어댑터 파일만 늘린다.
- 유료 사이트는 **개인용·저빈도·비재배포** 전제로 조심히 운영(요청 간격·캐싱). 약관 리스크 인지.
- **공매(onbid)** 는 번호 체계(물건관리번호)가 법원경매 사건번호와 달라 조회수 소스와의 매칭이 제한적일 수 있다. 공매는 기본정보·유찰 기반 지표 중심으로 가고, 조회수는 매칭되는 경우에만 붙인다.

## 4. 아키텍처

### 스택

Python / FastAPI(웹+API) · PostgreSQL(**Supabase 무료 티어, DB 전용 사용**) · Jinja2 + HTMX(대시보드) · 크롤 실행은 **GitHub Actions 예약 워크플로**.

### 구성요소

1. **소스 어댑터(플러그인)** — 사이트별 독립 모듈. 공통 인터페이스로 "물건 식별자 → 표준화된 결과(조회수/기본정보/매각결과)"를 반환. 어댑터 5종: `courtauction`, `onbid`, `ggi`, `tank`, `dooin`. 조회수 어댑터(ggi/tank/dooin)는 완료 사건의 **낙찰가율**도 함께 뽑도록 확장.
2. **크롤러 잡** — 독립 파이썬 스크립트. GitHub Actions cron이 실행 → 어댑터 호출 → Supabase에 스냅샷 저장. Supabase 접속키는 GitHub Actions Secrets에 보관.
3. **집계 엔진** — 스냅샷을 물건별로 합쳐 조회수 추세·지표·경쟁열기 점수·비교분석을 계산.
4. **웹앱/대시보드** — 단일 비밀번호 로그인 후 워치리스트·추세·점수·비교분석 표시. 물건 추가/삭제 UI. 스케줄러가 없어 무료 호스팅에서 잠들었다 깨어나도 무방(읽기 위주).

### 데이터 흐름

```
워치리스트 등록(사건번호/공매번호 입력)
   → [매일] 크롤러가 각 어댑터 호출: 조회수 + 상태(매각 여부)
   → [주1회/등록시] 유사 비교군 사건 + 매각결과 수집
   → 원천 스냅샷 저장(Supabase)
   → 집계 엔진: 추세·점수·비교분석 계산
   → 대시보드에서 확인(로그인 필요)
```

### 소스 간 물건 매칭

- 법원경매: **사건번호(예: 2024타경12345) + 물건번호**를 공통 키로 조회수 소스와 연결(유료 사이트가 사건번호 노출).
- 사건번호 정규화(공백·형식 차이 흡수) 로직 필요.
- 자동 매칭 실패 시 **수동 URL/ID 지정** 허용(`source_links.match_status = manual`).

## 6. 지표 및 경쟁열기 점수

### A. 사전 신호(워치리스트 물건, 실시간)
1. **조회수 추세(핵심)** — 소스 합산 조회수 + **일별 증가속도(views/day)**. 절대 조회수는 등록 경과일에 좌우되므로 최근 증가 속도와 **개찰 임박 시 급증 여부**를 중시.
2. **최저가/감정가 비율** — 유찰로 싸질수록 관심 몰림.
3. **유찰횟수** — 양면 신호(수요 약함 vs 저가 매력). 값만 표시, 해석은 사용자.
4. **개찰일까지 남은 일수** — 임박도.

### B. 실증 신호(비교군 기준)
5. **비교군 낙찰가율 분포**(중앙값·사분위) → 예상 낙찰가 밴드 + 경쟁 강도.
6. **비교군 매각 성사율**(매각/전체) → 이 조건 물건이 잘 팔리는지.
7. **비교군 평균 유찰횟수·매각 소요기간.**

### 유사(비교군) 매칭 기준 — 조정 가능 파라미터
- 같은 **용도**(property_type) — 필수
- 같거나 인접 **지역**(관할법원/시군구)
- **감정가 ±X% 밴드**(기본값 예: ±20%)
- **유찰횟수 ±1**
- **최근 N개월 내 매각**(기본값 예: 12개월)

### 종합 "경쟁열기 점수" — A + B 조합
- **A. 투명 가중합 지수** — 위 신호들을 0~1로 정규화 후 가중합 → **0~100 점수**. 가중치 조정 가능. 항상 **구성요소 분해표**를 함께 표시(블랙박스 금지).
- **B. 백분위** — 물건의 조회수 속도·예상 낙찰가율이 **비교군 내에서 몇 %인지**(예: "조회수 속도 상위 20%").
- 대시보드: 가중합 점수(크게) + 비교군 대비 백분위 배지 + 구성요소 표 + **"보조지표(예측 단정 아님)" 문구**.

## 7. 수집 주기

- **매일(KST 정해진 시각 1회, GitHub Actions cron):** 워치리스트 물건의 조회수·상태(매각 여부) — 시간에 민감한 신호.
- **주 1회 또는 물건 등록 시:** 유사 비교군 사건 + 매각결과 수집 — 완료 데이터라 변동 적음.

## 8. 접근 제어(인증)

- **단일 비밀번호 + 세션 로그인.** 비밀번호는 해시로 저장(예: bcrypt/argon2), 세션 쿠키, **HTTPS 전제**.
- Supabase Auth는 사용하지 않음(앱 레벨 단일 비밀번호가 더 단순·이식성 우위).

## 9. 크롤 운영 / 에러 처리

- **차단·부하 완화:** 요청 간 랜덤 지연, 표준 User-Agent, 소스별 동시성 제한. 매일 잡은 워치리스트 수만큼만(가벼움).
- **부분 성공 허용:** 실패 시 지수 백오프 재시도. 한 소스 실패가 전체 수집을 막지 않음.
- **파싱 취약성 대응:** 파싱 실패 감지 시 `match_status=unmatched`/에러 로그 후 대시보드에 "수집 실패" 표시(조용히 틀린 값 축적 방지). `raw`(jsonb) 원본 보관 → 파서 수정 후 재계산.

## 10. 테스트 전략

- **어댑터:** 저장한 샘플 HTML 픽스처로 파싱 단위테스트(실사이트 호출 없이 회귀 방지).
- **집계·점수 로직:** 합성 스냅샷으로 단위테스트(조회수 속도·낙찰가율 분포·가중합·백분위 계산 검증).
- **매칭 로직:** 사건번호 정규화(공백·형식 차이) 테스트.

## 11. 미결 구현 디테일(구현 초기 결정)

- **웹앱 무료 호스팅** 선택(Render / Railway / Fly.io 중). 읽기 위주 대시보드라 어디든 무방 — 배포 시 결정.
- **각 소스의 조회수/결과가 실제로 로그인 없이 보이는지** 실제 페이지로 검증 후 확정(어댑터 구현 초기).
- 유사 비교군 수집을 위한 courtauction/onbid **검색(필터) 크롤 방식** 구체화(용도·지역·감정가 밴드 필터 파라미터 매핑).

## 12. 향후 확장(본 스펙 범위 밖)

- 축적된 `comparable_cases` + `auction_results`(낙찰가율 등)를 학습 데이터로 한 **예측 모델(B)**.
- 조회수 소스 어댑터 추가(옥션원·스피드옥션 등).
- 조건 필터 기반 워치리스트 자동 편입.
